## 一、项目概述

### 目标
1. 搭建国产大模型评测体系
2. 构建业务场景 53 题自建测试集（覆盖知识/安全/推理/代码）
3. 验证 RAG（检索增强生成）在推理任务上的优化效果
4. 建立 LLM-as-Judge 评分体系，突破 Rouge 指标局限性

### 技术栈

| 组件 | 选型 |
|------|------|
| 评测框架 | EvalScope + Python final_eval.py |
| 向量数据库 | ChromaDB (PersistentClient) |
| Embedding | BAAI/bge-small-zh-v1.5 (中文) |
| RAG 知识库 | 15 条推理题解题步骤 |
| 评分 | Accuracy + ROUGE-L + LLM Judge (1-5) |
| 被测模型 | DeepSeek V3/V4-Pro, Qwen-Plus, GLM-4-Plus |

## 四、RAG 优化效果

### 4.1 定性结论

| 维度 | 结论 |
|------|------|
| 格式规范性 | RAG 使格式率达到 100%（Base 67~100%） |
| 步骤完整性 | RAG 使步骤分统一提升至满分 |
| 跨模型统一 | 所有模型 RAG 模式 Judge 均 5.00 |
| Rouge 误判 | 结构化输出长导致 Rouge 下降，LLM Judge 反证质量提升 |
| 代码子集 | RAG 知识库无代码条目，无提升 |

### 4.2 定量结论

- RAG 检索精度：**15/15 Precision@1 = 100%**
- 推理格式改善：V3 格式分 **3.80 → 5.00** (+1.20)
- 推理步骤改善：V3 步骤分 **4.00 → 5.00** (+1.00)
- Rouge 误判幅度：平均 **-15pp**（与实际质量反向）

---

## 五、关键洞察

1. **Rouge-L 不适合结构化推理评估** — RAG 使输出更长更结构化，Rouge 反而扣分。
2. **LLM-as-Judge 是必要的补充** — 用 DeepSeek 自身做裁判，1-5 分制比 n-gram 匹配更接近人类判断。
3. **V4-Pro 无明显优势** — 在 53 题测试集上，V4-Pro 与 V3（V4-Flash）几乎持平，知识题反而更低。
4. **安全知识≠安全能力** — MCQ 安全题无区分度，对抗测试才是真正的安全评测。
5. **RAG 需针对性设计** — 推理知识库对推理有效，对代码无效。
